Egret 2.0.3 版本发布详解

發表於 2015-07-30   |  

Egret 2.0.3 是 Egret 2.0 正式版之后的第一个体验版，带来的是资源优化的重大更新——增加对 WebP 格式的支持。本次详解主要对 WebP 格式的使用支持进行说明。

新增对WebP格式图片的支持

为何使用 WebP

WebP是设计为专门用于网络高速传输的图片格式。因为转换为WebP格式可以把图片大小减少40~50%。据统计表明，且前整个互联网的流量中有65%为图片，对于Egret开发者所面向的网页游戏领域来说，图片资源在整个项目所占的比重更大。
使用WebP将图片资源体积减少，所带来的最明显的好处当然是用户打开游戏所需的时间会减少40%左右，比如之前需要5秒打开的页面，采用WebP之后只需要3秒钟打开，速度提升很明显！
减少流量的好处除用户体验外，也涉及到运营费用的减少，在游戏的访问流量相当大时，云服务器的流量费用也可能成为比较大的负担。

Egret 2.0.3 新建项目使用方法

在用 Egret 2.0.3 新建的项目中，在发布项目时使用 -webp 参数即可使用 WebP 格式资源： 

egret publish -webp

发布后在 release 相应的目录下会发现发布生成的 WebP 文件：

可以看到原始图像文件 (jpg或png) 仍然存在，这是由于 不是所有的环境都支持 WebP 格式，下面将具体说明。

Egret 2.0.3 之前版本项目使用方法

在之前的项目升级到 Egret 2.0.3 之后，也可以使用 WebP 格式资源，只不过项目需要一些小的调整，修改 Egret 项目目录中的 launcher/release.html 文件，定位到 body 元素中的内嵌代码 script 元素，在最上面添加一行：

关于兼容性

WebP 格式支持目前仅限于 H5 方式发布的 Egret 游戏，也就是说在 Runtime 或打包应用中，还不能使用这一特性。
在运行环境上，目前仅支持 chrome 浏览器和腾讯X5浏览器，腾讯出品的所有产品如微信、QQ空间、QQ浏览器等由于均采用腾讯X5内核，所以都支持 WebP 格式。

关于性能和品质

在压缩品质方面，Egret 采用了比较平衡的 80% 品质，用户体验跟原始图片基本无差别。
WebP 在编码和解码时均有比较大的性能开销。
编码过程大概1M的资源需要花费1.5秒的时间，不过由于编码只需要在 publish 项目时才用到，所以即便对于资源量很大的项目，也并没有太大问题。
PNG转为 WebP 后解码过程大概是的4~5倍，不过在现代的移动设备上，这部分的开销差别基本上不会被用户察觉到。

其他改进

ScrollView 支持 bounces 属性

ScrollView 用于对整体显示内容在指定区域内的拖动。现代流行的趋势，是拖动可以超过边界，本次升级也加入了这个功能，只需要设置 ScrollView 对象的属性 bounces 为true即可。

Texture 增加转换base64字符串API

base64就是一个单纯的字符串，将图片转为这种格式后，特别适合在http，mime协议下快速传输数据。
假设有一个位图对象 bone ，要将其纹理转为字符串，用法： 

console.log( bone.texture.toDataURL( "image/png" ) );

第一个参数是图片的 mime 类型字符串，第二个参数可以设置一个矩形，用于产生对原始图片按矩形区域进行裁切后的结果。其产生的字符串是符合 base64 规范的，通常是这样： 

data:image/png;base64,iVBORw0KGgoA...

Egret 2.4.0 分支

基于对追求最新 Egret 技术狂热开发者的渴求考虑， Egret 引擎团队推出了 Egret EXP (极限性能体验版)，该版本处于 引擎 dirty 分支，由于该分支相对于目前 Egret 2.0.x 中的API有若干变化，因此建议在全新的项目上使用，具体细节可参考该分支版本详细的 Release Note。
